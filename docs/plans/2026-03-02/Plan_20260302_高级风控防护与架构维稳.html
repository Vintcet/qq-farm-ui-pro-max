🚀 Plan: 高级风控防护与架构维稳 (Anti-Risk & Stabilization)

创建日期: 2026-03-02 | 目标: 修复近期架构隐患并引入仿生学安全护城河。

1. 核心挑战与背景

随着系统向 MySQL + Redis 架构的全面演进以及高精度 “蹲守抢收” 等功能的引入，系统从过去的“被动定时访问”演变成了高强度的“精准执行并发矩阵”。在大幅度提高收益的同时，面临着以下双重挑战：

  • 底层架构耗损低配服务器长连接休眠断链与 CPU 事件循环组（Event Loop）拥堵。
  • 行为特征僵化系统过高的“精准率”和“绝对不掉线”的非人特性，极易被大厂的大数据模型标记为特征级脚本。

2. 已知风险治理 (Impact Assessment & Remediation)

2.1 MySQL 连接池断链与老化 (ECONNRESET)

影响面分析： 深夜无人操作或云厂商为了释放资源强杀空闲 TCP 连接后，mysql2 线程池内的连接变为僵尸连接。早晨唤醒瞬间，第一个获取到该句柄的 Worker 会抛出底层 Socket 断开错误。

修复对策 - 「活性保活与探活重试机制」：

  • 在 MySQL 连接池初始化时，追加 enableKeepAlive: true 与 keepAliveInitialDelay: 10000 参数，维持底层 TCP 心跳。
  • 给所有的数据库操作包裹一层重试闭包 (Retry Wrapper)。假如捕获到 ECONNRESET 或 PROTOCOL_CONNECTION_LOST，主动强制清理该挂掉的链接并重试一次，对上层业务 100% 透明。

2.2 低配服务器 Event Loop (事务循环) 队列拥堵

影响面分析： 开启“蹲守机制”后，每个账号衍生出高度并行的微观倒计时任务。多账号同时激活时，V8 引擎因为 CPU 分配不均，造成毫秒级级别的延迟失效，导致 60 秒的黄金抢收卡点失败。

修复对策 - 「分片任务与时间补偿」：

  • 不再信赖 setTimeout 的绝对准时，重构调度器（Scheduler），采用 时间补偿差 (Time Compensation) 算法补平 CPU 挂起的时间丢失。
  • 削减不必要的并发数组遍历映射（Map/Filter 等同步大循环），加入 setImmediate 或者微小 sleep(10) 把重量级的 JSON/数组比对过程让出主线程切片。

3. 下世代仿生风控防御体系 (Next-Gen Anti-Risk)

为将脚本伪装成接近物理人效的操作，需将以下三级智能随机化机制作为接下来的核心主线开发规划。

3.1 随机“打盹/掉线”机制 (Micro-Sleep / Ghosting)

graph TD A[账户定时巡查轮询] --> B{进入防沉迷判定} B -- 触发(小概率) --> C[强制休眠 30-90 分钟] B -- 未触发 --> D[正常执行巡查] C --> E[产生自然人离线行为断层] E --> F[苏醒后追加一次密集收获] D --> G[结束]

架构设计：引入账户级的体力疲劳模型。工作越久，触发 Hibernate(挂起) 的概率增加。在挂机界面可视化展示“休息中”，强行断绝与腾讯服务器的一切 API 牵连。

3.2 组网错峰唤醒 (Jittered Wake-up)

打破现阶段在早晨 07:00（静默时段结束）所有账号一拥而上的请求脉冲图谱。

  • 计算基准：以账号 userId / Gid 为基准计算一个固定的哈希散列时间（范围：0 ~ 45 分钟）。
  • 错峰执行：例如系统静默于 07:00 结束。A 账号哈希加值为 12 分钟，其实际启动时间为 07:12。B 账号为 07:23 启动，彻底抚平每日启动峰值的特征。

3.3 “无效”操作特征混淆 (Noise Injection)

  • 人类玩游戏有很多是纯粹的“走走看看”，而脚本每次请求都一定带有极高转化率的成果反馈。
  • 实施方案：向好友轮询操作注入极小概率的“只看不动”请求，或者随意拉取一下商城的货物数据单页，混淆腾讯用来评估异常转化率的统计表表头。
