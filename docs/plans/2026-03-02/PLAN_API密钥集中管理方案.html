第三方 API 密钥集中管理方案

Architecture Design & Impact Analysis Document

1. 需求共识 (Goal & Consensus)

业务目标：将现散落在代码和环境变量中的第三方业务 API Key 集中管理，实现“控制台化”配置。避免新环境部署时每次都需要修改代码或配置死板的环境变量。
  • 纳管范围：微信扫码相关 API (wxApiKey, wxAppId, wxApiUrl)。这是目前涉及第三方交互的核心业务密钥。
  • 排除范围：明确排除通知模块 (用户各自配置 的 Webhook/Push Token) 以及 Coding AI 相关配置 (DASHSCOPE_API_KEY, OPENVIKING_URL)。
  • 界面入口：在超级管理员界面（Settings.vue）新增配置面板，仅限 role: 'admin' 修改。

2. 架构设计 (Architecture)

后端持久化层

沿用现有的 store.js 及其 globalConfig 体系。扩展结构，新增 thirdPartyApi 节点，数据持久化存储至 store.json。

前端控制台

在现有的 Settings.vue (系统设置面板) 增加一个专属的「第三方 API 设置」卡片。结合 setting.ts Pinia Store 实现数据的无缝双向绑定。

3. 影响分析 (Impact Analysis)

本次改造主要涉及 Node.js 后端配置存储层、API 控制器以及对应的前端 Vue 组件，不涉及底层核心挂机引擎的运行逻辑，安全性与隔离性高。

文件路径 变更类型 影响细节 (Details)
core/src/config/config.js MODIFY 移除 wxApiKey, wxApiUrl, wxAppId 等硬编码，作为初始化的降级 Fallback 默认值提供。
core/src/models/store.js ADD 引入 DEFAULT_THIRD_PARTY_API_CONFIG，在全局对象 globalConfig 增加 thirdPartyApi 领域，并提供 get/set 以及 normalize 函数，触发 json 刷盘。
core/src/controllers/admin.js MODIFY 新增 GET /api/admin/third-party-api 和 POST /api/admin/third-party-api 路由。
将调用微信 API 进行扫码和确认等阶段的 CONFIG.wxApiKey 替换为从 store.getThirdPartyApiConfig() 读取动态配置。
web/src/stores/setting.ts MODIFY 新增拉取和保存第三方 API 配置的 fetch/save thirdPartyApiConfig 接口对接函数。
web/src/views/Settings.vue MODIFY 仅当 isAdmin 为 true 时，展示"第三方API设置"专属区域面板。
技术红线校验：
  • 由于微信二维码获取强依赖 admin.js，动态替换 API 参数后需要充分测试二维码拉取与验证的回调通路是否畅通。
  • 为了安全，前端只有获取到带有 admin role token 的超级管理员，后端才会吐出完整 API Key，这符合现有的 RBAC 防越权体系。

4. 下一步行动纲领 (Next Steps)

基于当前计划，我们将分为三个实施阶段：

  • Step A (后端数据层改造)： 增加 Store 定义，完成配置读持久化及对外暴露的管理路由。
  • Step B (后端调用层改造)： 去除微信扫码模块硬编码的依赖，由配置中心动态赋能。
  • Step C (全栈打通及 UI 呈现)： 编写 Pinia Store 方法及 Settings.vue 无缝交互表单，联调验收。
