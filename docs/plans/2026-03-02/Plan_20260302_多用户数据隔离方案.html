🚨 系统多用户数据隔离专项修复方案

背景说明：当前系统在集成体验卡（多用户）后，仍然呈现“单租户”的数据表现。无论登录哪个账号，都能看到所有的微信和 QQ 账号数据，且可以对其进行各类游戏/自动化操作。此缺陷严重破坏了系统的数据安全性和多用户可用性，需要立即修复。

一、 现状缺陷诊断 (Root Cause Analysis)

基于对 core/src/controllers/admin.js 和 core/src/models/store.js 的全面源码审查，确认以下 3 个核心隔离失效点：

  1. 源头缺失（创建与数据存储层）：

    在 store.js 的 addOrUpdateAccount(acc) 中，新创建账号时，硬编码了写入字段（id, name, code, platform, uin, qq, avatar），没有保存所有者的 username。导致底层的 store.json 账号列表毫无归属标记。

  2. 列表泄漏（读列表层）：

    在 GET /api/accounts 端点中，直接调用 provider.getAccounts() 并原样返回了全部数组，没有根据 req.currentUser.username 进行过滤。

  3. 越权操作（写操作与详情读取层）：

    虽然开发者在 admin.js 预写了 accountOwnershipRequired 中间件用于验证权限，但在所有涉及到单账号操作的路由上都没有挂载该中间件！任何人只要拦截或枚举 x-account-id 获取到了别人的账号 ID，就可以堂而皇之地调用 /api/automation、/api/farm/operate 甚至进行一键删除偷菜等高危操作。

二、 架构改造方案 (Architecture Design)

三、 原子执行拆解 (Atomized Tasks)

Step 1: 修复底层模型，保障 Username 的持久化 (Store 改造)

  • 目标：新增账号时打上数据标签。
  • 方案：
    1. store.js 中的 addOrUpdateAccount 修改传入参数，解析提取 acc.username。
    2. 数据组装 data.accounts.push 时补充存入 username: acc.username || ''。

Step 2: 修复控制器，挂载全局所有权校验拦截 (Admin Controller 改造 1)

  • 目标：拦截恶意 ID 操作请求。
  • 方案：
    修改 admin.js 中的中间件 accountOwnershipRequired，增强判读逻辑： 
    // 如果账号旧记录没有 username 怎么办？(向下兼容)
// 结论：没有 username 的统一视为 "admin" 的资产。
if (!account.username && req.currentUser.role !== 'admin') {
    return res.status(403).json({ ok: false, error: '无权操作系统公共账号' });
}

    将中间件串联到**所有**具有用户强相关性的 API 上（如下）：
    • GET /api/status, GET /api/lands, GET /api/friends, GET /api/settings
    • POST /api/automation, POST /api/settings/save
    • POST /api/accounts/:id/start|stop
    • DELETE /api/accounts/:id
    • POST /api/farm/operate

Step 3: 修复列表读取，实现视图数据隔离 (Admin Controller 改造 2)

  • 目标：在 GET /api/accounts 中进行数据清洗。
  • 方案： 
    app.get('/api/accounts', (req, res) => {
    let data = provider.getAccounts();
    // 数据深度拷贝以避免污染缓存引用
    let accountList = [...data.accounts];
    // 数据清洗过滤
    if (req.currentUser && req.currentUser.role !== 'admin') {
         accountList = accountList.filter(a => a.username === req.currentUser.username);
    }
    res.json({ ok: true, data: { ...data, accounts: accountList } });
});

Step 4: 保存帐号时的用户名植入 (Admin Controller 改造 3)

  • 目标：通过 API 创建账号时植入所有权归属者。
  • 方案： 在 POST /api/accounts 端点中： 
    const payload = isUpdate ? { ...body, id: resolvedUpdateId } : body;
// 强制注入数据归属者
if (!isUpdate && req.currentUser) {
    if (req.currentUser.role !== 'admin') {
        payload.username = req.currentUser.username;
    } else {
        // admin创建的如果不特别指派，可以留给admin
        payload.username = req.currentUser.username; 
    }
}

四、 验收标准 (Acceptance Criteria)

  • ✅ 管理员通览：使用 admin 账号登录，账户列表中可正常看见所有账户并正常控制启停。
  • ✅ 数据隔离：使用体验卡账号 userA 登录，列表只会显示 userA 自己绑定的号。
  • ✅ 拦截测试：尝试使用 userA 的 Token 调用系统里别的账户ID 的 API （如获取状态、保存设置），应当精准被 403 无权操作 拒绝拦截。
