🚀 深度审计与架构复盘报告 (2026-03-02)

这份文档回顾了系统近期的一系列重构（从 v3.6.0 到 v3.8.2 的过渡期），审查了目前落地的功能对整套架构的全局影响，并提出后续预防雪崩的建议。

1. 近期核心优化里程碑 (What we did)

1.1 存储与并网架构 (MySQL + Redis 化)

已完成 **行为**：从 SQLite 单线程写入剥离为 MySQL 8.0 持久化 + Redis 高速读取。`auth_data` 的授权指纹进行了数据库层级独立落盘封装。

**正面影响**：突破了 SQLite 处理多账号轮询时常见的 database is locked 和 `busy` 问题；通过全量替换 Promise API，底层不再会有阻塞调用（如 `fs.readFileSync` 或 `JSON.parse`）堵死 Node.js 的主线程 Event Loop。

1.2 鉴权长连接与防 400 死亡死锁

已完成 **行为**：修复了 [WS] 400 错误，重写了 `startAccount` 加载流程使其具备 `openID` 透传，并在前端/后端加入了长达 15s 的断连重连防抖拦截。

**正面影响**：极大缓冲了 WebSocket 由于腾讯服务器强行断开导致的无效握手与递归请求风暴（防止机器自动高频重连直至被永久封锁 IP）。

1.3 Jitter 排队与 1002003 风控软着陆

已完成 **行为**：新增 `suspendUntil`，在 `network.js` 检测到频控报错时直接掐断并强制休眠 30 分钟，此外应用内所有延时和单发操作全部加入了指数退避（Exponential Backoff）和 Jitter 随机抖动。

**正面影响**：避免群发外挂的机械特征被服务端大数据分析锁定；从技术上斩断了无限抛错造成的 CPU 空转资源浪费。

2. 隐患分析与副作用排查 (Impact Analysis)

🔥 隐患 1：JSON 反序列化依然存在内存尖刺 (High Risk)

问题源 触发条件 可能导致的后果
store.js 中的 getAccountsFull() 当运行上万账号，管理员点击“一键全启动” 尽管解决了 MySQL 的高频 `Select` 请求，但 findAll() 返回的一万条原生记录会在 Node.js 主线程中执行一万次 JSON.parse(r.auth_data)。这将造成超过数百毫秒甚至一秒的主线程绝对停顿（Event Loop Block），导致在此期间的所有其他请求（如心跳判定）全部挂起，甚至引起其它账号掉线。

**建议方案**：如果日后拓展至万级别账号，`getAccountsFull()` 依然不能“一板斧加载”。建议 `runtime-engine.js` 中的 startAllAccounts 改用**分页启动** (如：每次查询 100 条并启动，然后 `setTimeout` 等待 500ms 再查下 100 条)。

🔥 隐患 2：风控自愈锁缺乏持久性记录 (Medium Risk)

问题源 触发条件 可能导致的后果
utils/network.js (suspendUntil) 账号被风控 (1002003)，并且此时进程刚好被 PM2 或 Docker 重启 现在的 `suspendUntil` 写定在 const userState = {} 的内存堆中。一旦服务器主进程遇到更新、崩溃重启，这 30 分钟的静默记忆就会灰飞烟灭。紧接着程序启动又会直接撞上腾讯的枪口并反复吃满 1002003，进一步加深封禁强度。

**建议方案**：将 `suspendUntil` 这个封控时间戳持久化到 Redis 或者 MySQL 的 `status` 字段中。在 `startAccount` 拉取时读取，如果还没解封就干脆不要创建 WebSocket 去骚扰腾讯网关。

🔥 隐患 3：僵尸任务的定时器内存泄漏 (Low/Medium Risk)

问题源 触发条件 可能导致的后果
scheduler.js / network.js 反复触发断线重连或手动反复启动停止账号 目前虽然设计了 `cleanup()`，但如果我们用 setTimeoutTask 发出的待办如果在闭环中未被严格取消，积累多了会导致调度闭包 (Closures) 抓着大量的 `userState` 上下文无法被 V8 垃圾回收 (Garbage Collector) 给回收。

4. 下一步建设建议 (Next Steps)

  • 安全层面：进一步隔离微信通讯接口密钥，防止 `admin.js` 还残留任何未加密的明文代理凭据并提交至公网。
  • 多进程架构 (Cluster)：Node.js 单线程模型在承载 500+ WS 链接时就会面临瓶颈。在下周期的计划里，我们需要将 runtime-engine 转换为 Worker Thread 或独立 PM2 子进程。
  • 前端状态下发：当前 `suspendUntil` 锁定后，前端用户看面板依然是显示“运行中”。应当通过 WebSocket 下发一个 { event: 'account_suspend', accountId, expireAt } 状态到前台，给用户展示“☕风控冷却中”的 UI 样式以免误会。
