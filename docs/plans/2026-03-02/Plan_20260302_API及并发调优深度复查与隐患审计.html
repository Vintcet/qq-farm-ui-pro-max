1. 近期核心改动回顾与健壮性评估

✅ 第三方 API 密钥控制台化

状态：已平稳落地。

逻辑闭环完善，如果在面板留空，后端依然有 CONFIG.wx... 原生配置作为终极回退锚点，不会发生异常 Undefined 崩溃。此举也顺道解除了对环境变量配置重载依赖的心智负担。

✅ 并发启动与 JSON解析 阻塞处理

状态：雪崩防御生效。

getAccountsFullPaged 切片与 600ms 的平滑注入，有效压平了 V8 引擎解析宏大数组库时造成的 CPU 阻塞。即便是 1000 个账号，系统也能在稳定低负载下分发就绪。

⚠️ 全局回退至 “极度保守预设”

状态：有效规避封号，业务效益降低。

所有未声明新配置（即老数据或初建号）都会降级为 300秒-600秒，外加夜间静默。对于主号求稳是天作之合，但如果是专业羊毛党可能觉得效率断崖级下跌（需告知用户到新面板里一键应用“激进”预设）。这是有意为之的 Trade-off（妥协）。

2. 潜在的新隐患与改进建议 (Impact & Risks)

基于全局代码的交叉引用，发现了几处未来（特别是云平台下）可能引发瓶颈和数据冲撞的点：

隐患 A：风控标记并发导致的 Redis / DB 集群写入踩踏

背景机制：当腾讯抛出 `1002003`（访问频繁）错误时，我们会执行 recordSuspendUntil。若发生大规模集群惩罚（如腾讯拉闸拦截单个IP），那一瞬间可能有上百个挂机账号同时进入 `Network.js wsError`。

风暴点：他们会同时调用 setAccountConfigSnapshot。尽管底下套了防抖 debounce 批量写机制，但在极端的宏任务中内存队列的 `JSON.stringify` 仍会引发抖动。

优化建议（Phase 4）：将 runtimeRecords（包含 `suspendUntil` 等高度频繁变动的数据）完全从主 store.json / SQLite / MySQL 全量 Config 体 中剥离，只存在高速 Redis 或专门的状态表里。

隐患 B：Vue 的 ThirdPartyApi 配置没有多端冲突仲裁

背景机制：之前的 UI 设置（如黑夜模式、极简模式）我们增加了 `uiTimestamp` 来防止多端冲刷。但新加的 thirdPartyApi 目前并未纳入该时间戳防护。

影响面：鉴于这是一个仅限 role: 'admin' 的超级管理员操作，出现两个超管同屏抢占修改密钥的可能性极低，但在理论层面仍是无锁盲区。

优化建议：目前可无视。日后可在 saveGlobalConfig 增加全量版本号乐观锁（Revision Lock）。

隐患 C：Node.js WebSockets 客户端无心跳被动卡死

背景机制：目前遇到腾讯 API 异常能够熔断，但如果处于 TCP 半开连接状态（如 Docker 桥接异常、主机休眠、断网等），底层 WebSocket 的 on('error') 和 on('close') 是**不会被触发**的。它会像僵尸一样一直悬挂。

优化建议：在 network.js 里补上一个客户端主动 Ping/Pong 心跳探活机制。如果在 CONFIG.heartbeatInterval 的 2.5倍 内没收到任意合法数据帧，强制调用 ws.terminate() 摧毁尸体并触发我们完善的垃圾回收及重连逻辑。
