🚀 农场核心：高并发与风控自愈架构优化 (Phase 2)

文档版本: v1.0 | 日期: 2026-03-02 | 阶段: Plan (设计与规划)

背景综述：近期在对底层 MySQL 通讯、多开隔离进行了深度改造，修复了 400 授权掉签等严重事故。但随着业务复杂度和承载目标（千/万级账号）的攀升，并发启动时的 I/O 雪崩危机、腾讯服务端对“代挂机”行为的风控（如 "1002003 假封禁" 拦截），以及高频循环引发的僵死问题，共同构成了下一阶段的“三座大山”。

一、数据库并发回查重构 (防启动雪崩)

解决海量并发场景下 startAccount 中 for-loop 枚举查 getAccountFull 导致的 API 积压与阻塞。

1. 痛点分析

如果在页面触发“全部启动”，1000 个账号会向 mysql-db 连接池暴雨般倾泻 1000 次 SELECT * FROM accounts WHERE id = ? 的点查。极易导致死锁、内存飙升并拖垮接口响应。

2. 解决路径

  • 方案 A（内存级缓存投影）：在执行 findAllLite 获取轻量列表时，将 code 和 uin 剥离进一个独立于主大 JSON (auth_data) 之外的、能被轻量查询直接提取的新列。这需要修改数据库字段 Schema。
  • 方案 B（SQL IN 批量化预提）：重写批量启动入口。在准备启动 1000 个账号时，收集所以 ID，在进入 worker-manager 分发前，执行一次 SELECT id, auth_data FROM accounts WHERE id IN (...)，然后通过 Map 映射高速匹配。 (建议采用此方案，无需动表结构，效果立竿见影)

二、腾讯防风控拦截反制解析 ("1002003 假封禁" 风波)

解决高频请求被腾讯误判并抛出 code=1002003 账号已被封禁 的风险拦截。

风险剖析：腾讯为何报“封禁”却没真封？
由于农场端本质上是一个 HTML5 游戏。当后台 Worker 每分每秒都在不停发出 enter_farm 或偷菜时，在腾讯的审计大脑（如 WAF）看来，这个 IP 发出的频率和时序“根本不像物理人类”。
所谓 1002003 实际上是服务端级别的软限制（Rate Limit / Soft Ban），系统把你当“异常机器”屏蔽了这部分下行数据的回流。

反制护城河架构

graph TD A[Worker 发起拦截点] --> B{请求频率分析仪} B -->|短时过于密集| C[熔断器介入触发] C --> D[注入拟人化随机延迟 Jitter] B -->|正常频率| E[发往腾讯网关] E --> |报 1002003 假封禁| F[执行指数退避休眠 Exponential Backoff] F --> |沉默 5 - 30 分钟后| A
  • 全局动态速率节流机制 (Global Rate Throttler)：不要让程序“无脑”执行。巡田、好友互动的间隔要加入 `Random(10, 60)` 秒的浮动扰动（Jitter），禁止绝对精准的机械化秒级发包。
  • 识别假装与软着陆：在底层截获 1002003，不要打出“已被封禁”这种引发恐慌的 Error。而是映射为 "遭遇高频风控限制"。将该账号挂起（Suspended），强制使它休息 30 - 60 分钟后再重试，绝不要连续撞墙！

三、全量熔断与自退避机制 (死循环清理)

将最近成功验证的“报错即掐断”防抖思维普及至整个农场架构。

实施规范

  • 审查所有的 while(true)： 诸如长轮询监听收割、网络断连复充等模块，若遭遇连续 3 次以上的网络 Reject 或非 200 错误：
    1. 不再立即 await sleep(1000) 然后原地复读机。
    2. 立刻上锁进入 指数退避（Exponential Backoff）：失败1次等 5s，2次等 15s，3次等 60s，以此类推。
  • 看门狗 (Watchdog) 探针： 只要发现某个挂机账号的进程出现 Event Loop Lag，主动发送 STOP 指令给自身，在日志流中报告“因安全原因被系统休眠”。

四、影响面分析 (Impact Analysis)

  • 对数据库的影响：采用方案 B 大幅降低连接池高并发的抢占。启动耗时将由原先的 O(n) 变成了 O(1)，对服务器内存极为友好。
  • 对偷菜效率的影响：由于引入了防封禁抖动（Jitter），可能偶尔会导致无法绝对卡在第 0 秒抢收到好友资源（但此前的 60秒极限抢收本身就加入了时间提前量，所以自身逻辑不易受损）。但换来的是**封号概率直线收缩，挂机的存活率与安全性大幅跃升**。
