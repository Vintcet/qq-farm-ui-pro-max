一、背景与目标 (Background & Goals)

随着 QQ 农场智能助手逐渐发展为高颜值的现代化架构，前台交互数据的状态复杂度和体积与日俱增。本项目旨在完成最后两块体验体验飞升的拼图：

  • 状态一致性 (UI Sync Arbitration)：解决多设备、多标签页引发的状态争用和冲突覆盖难题。
  • 加载性能 (Asset Chunking)：对长链、重型组件实施切片按需加载（Lazy Loading），防止包体积随业务增长而崩塌。

三、原子化拆解任务 (Atomic Tasks)

Task 1 端云极速仲裁机制 (状态拉扯解决)

区域 动作描述
Web 修改 stores/app.ts，在保存 colorTheme 等属性时，附加一个 timestamp 并通过 useStorage('app_ui_sync_time') 记录。
Web 修改 api/auth/me 或登录后获取配置的回调钩子。如果发现后端下发的 uiConfig.timestamp 大于本机的 timestamp，才执行覆盖变更。
Core 修改 store.js 及 data-provider.js，在 setUITheme 数据结构中落库记录该时间戳。若无时间戳则视为强制覆盖。

Task 2 懒加载与代码切片 (包体积优化)

区域 动作描述
Web 将现有的非首屏必需品（如 NotificationModal.vue、StealSettings.vue）改为通过 defineAsyncComponent 导入。
Web 配置 vite.config.ts 中的 manualChunks（如果有必要），将 vendor (vue / vue-router / pinia) 大体积核心件锁定，业务层实现自然分割（.js.gz 碎片化）。
Web 引入一层异步加载期的占位 (Loading / Skeleton)，确保网速慢时用户感知平滑不留白。

四、影响面与风险评估 (Impact & Risks)

潜在风险点 缓解预案 (Mitigation)
浏览器本地时间不准 在首次鉴权接口获取服务器提供的绝对基准时间差，统一使用 Date.now() + offset 作为防伪时间戳。
按需加载失败(网络闪断) Vue defineAsyncComponent 自带容错参数设置。可以配备 errorComponent 或者重试闭包函数进行防御。
状态无缝合并的类型安全 TS 接口需允许 timestamp 为 Optional (向后兼容旧账本中无时间戳的用户)。
