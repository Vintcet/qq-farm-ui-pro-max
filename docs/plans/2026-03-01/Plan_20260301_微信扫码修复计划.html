🔍 故障原因分析

🧨 API 参数形式错误

现象说明

在先前的代码中，我们把 api_key 和 action 参数统统打包放在了 HTTP POST 请求的 body: JSON.stringify({...}) 中发给了 api.aineishe.com。而第三方接口网关因为在其路由 URL（Query String）中提取不到相关的 action 和鉴权字段 api_key，因此它被误判为一次“无状态/未被授权”的错误访问，下发了 “请先登录” 或者空白拒绝的回调信息。

对比 vxcode 参照代码: 在参照代码中，请求是通过 `${API_URL}?api_key=${encodeURIComponent(API_KEY)}&action=${action}` 直接拼接到 URL 问号后作为 Query 参数传递的，而我们原本写入的逻辑全都在 request body 里。

📦 后端路由修改计划

✨ 代码重构

实施方案

  • 重构所有的 fetch 函数。针对 getqr, checkqr 和 jslogin，把 api_key 和 action 脱离出 JSON body，强行拼接为 URL。
  • body 里只保留第三方需要的额外逻辑参数（如 uuid 或 appid），其它验证握手均上缴到 URL 中。

修改参照对比

修改前 (JSON Body 传递) ❌ 修改后 (Query URL 传递) ✅
fetch(CONFIG.wxApiUrl, {
  body: JSON.stringify({ action: 'getqr', api_key: CONFIG.wxApiKey })
}) 		fetch(CONFIG.wxApiUrl + '?api_key=...' + '&action=getqr', {
  body: JSON.stringify({})
})

✅ 其他可行的优化策略

1. 安全断联（AbortController）兜底设计

当前咱们已经在代码中注入了基于 AbortController 的超时熔断，能够防止请求出现因参数错误而被外部网关挂死导致资源耗尽的可能，这个安全项无需二次改动。

2. 失败重试

在获取 jslogin 获取微信授权 Code 的最后时刻（此时用户已在手机端点击允许），如果正好遭遇网络丢包，可考虑前端增加一个「重新获取」的状态重置闭环。
